# Tieba Fish 产品路线图

## 1. 当前产品判断

这个插件现在已经具备产品雏形，但还处于 `可用 MVP` 阶段，还没有进入 `闭环完整` 阶段。

当前最有价值的 3 个差异化点已经成立：

- `低打扰阅读`
- `结构化贴吧数据`
- `老板模式`

当前最大的问题不是功能太少，而是下面 3 个闭环没有完全打通：

1. 首次使用闭环
2. 持续阅读闭环
3. 产品认知闭环

## 2. 当前阶段的优点

### 2.1 方向是对的

现在的主路径已经不是“在 VS Code 里开贴吧网页”，而是：

- 优先用 `aiotieba` 拉结构化数据
- 在 Webview 里重排版
- 浏览器只作为 fallback

这说明产品方向已经从“快捷入口”转向“贴吧阅读器”。

### 2.2 已经有继续演进的工程基础

当前工程已经有这些基础能力：

- 侧边栏入口和原生 TreeView 结构
- 论坛页 / 帖子页 Webview
- 账号凭据和 Cookie 安全存储
- `Python aiotieba bridge + 网页抓取 fallback` 双数据源
- 本地缓存
- 历史记录
- 老板模式

这意味着后面加功能不需要重搭底座。

## 3. 当前核心问题

### 3.1 首次使用门槛偏高

现在的最佳路径依赖：

- 本机有 Python
- Python 可执行路径正确
- `aiotieba` 依赖可用
- 用户知道如何配置 `BDUSS / STOKEN`

这对普通用户来说门槛偏高，产品内也还缺少明确的环境诊断和状态提示。

### 3.2 信息架构还不够稳

当前 `最新` 的真实含义是：

- 最近一次打开的关注吧的当前页快照

这个名字容易让用户误以为它是：

- 所有关注吧聚合流
- 实时更新流
- 未读流

命名和实际能力之间还有偏差。

### 3.3 阅读留存链路不完整

现在可以打开帖子，也有历史记录，但还没有真正形成：

- 继续上次阅读
- 回到上次楼层
- 标记未读 / 已读
- 稍后再看

用户能“看”，但不一定愿意长期用它作为固定入口。

### 3.4 产品边界还不够统一

现在存在一些边界不一致的问题：

- 文档和现状并不完全一致
- 收藏能力还在数据层里，但主侧栏已移除
- 部分命令偏工程化，偏用户价值的表达还不够统一

这些问题不会立刻让功能坏掉，但会影响后续版本判断。

## 4. 产品目标重申

这个插件不应该做成“完整贴吧客户端”，而应该稳定收敛到下面这个定位：

**给常驻 VS Code 的用户提供一个低打扰、可快速伪装、可持续阅读的贴吧阅读器。**

这个定位下，产品优先级应该是：

1. 让用户更少切去浏览器
2. 让用户更快找到要看的帖子
3. 让用户更容易接着上次继续看
4. 让用户在需要时可以迅速切换到安全状态

## 5. 推荐版本路线

## 5.1 V0.2 可用性版本

目标：解决“第一次装上能不能顺利用”的问题。

必须做：

- 环境诊断页
- 当前数据源状态提示
- 账号状态提示
- `最新` 命名或语义修正
- README 与实际产品同步

建议做：

- 启动时自检 Python / aiotieba / 账号状态
- 提供“如何配置账号”的引导页
- 数据源失败时给明确恢复建议，而不是只报错

验收标准：

- 用户第一次安装后，知道自己缺什么
- 用户知道现在是登录态还是匿名态
- 用户知道当前是结构化数据还是 fallback 路径

## 5.2 V0.3 留存版本

目标：解决“看过一次后，为什么第二次还愿意继续用”的问题。

必须做：

- 同步我关注的贴吧
- 继续阅读
- 滚动位置或楼层级恢复
- 稍后看 / 收藏重新定义
- 阅读状态记录

建议做：

- 帖子未读标记
- 最近常看吧排序
- 打开时优先回到最近阅读内容

验收标准：

- 用户不需要每次重新找帖子
- 用户可以很自然地接着上次继续看
- 关注吧列表不依赖纯手工维护

## 5.3 V0.4 差异化版本

目标：把这个插件做成“只在 VS Code 里才成立”的摸鱼产品。

必须做：

- 老板模式增强
- 快捷键阅读流
- 关键词静默过滤
- 更强的低存在感 UI 模式

建议做：

- 可配置伪装项目模板
- 恢复老板模式前的滚动位置
- 伪装状态下的假 Open Editors
- 自定义阅读快捷键

验收标准：

- 产品在“摸鱼”场景下明显优于浏览器
- 用户可以不依赖鼠标完成核心浏览
- 老板模式进入和恢复都足够稳

## 6. Backlog 优先级

## 6.1 P0

这些是当前最值得立即排期的项。

- `环境诊断`
  说明：检测 Python、`aiotieba`、账号凭据、当前数据源。
- `账号状态可见`
  说明：让用户知道是否配置了 `BDUSS / STOKEN`，当前是否走登录态。
- `最新` 重命名或重定义
  说明：避免认知偏差。
- `同步关注吧`
  说明：直接读取贴吧账号的关注列表。
- `继续阅读`
  说明：恢复最近帖子和页码，最好恢复到上次楼层或滚动位置。
- `文档和界面统一`
  说明：去掉 README、菜单、实际能力之间的不一致。

## 6.2 P1

这些会明显提升产品体验，但不是当前阻塞项。

- `稍后看`
  说明：把原来的收藏能力产品化。
- `未读 / 已读`
  说明：让聚合阅读更有持续价值。
- `关键词过滤`
  说明：过滤固定噪音贴。
- `状态栏入口`
  说明：补一个统一的全局操作入口。
- `帖子内导航快捷键`
  说明：上一页、下一页、只看楼主、切图、展开回复。
- `阅读来源提示`
  说明：显示来自缓存、bridge、fallback。

## 6.3 P2

这些可以做，但不应该现在优先。

- `更复杂的伪装模板`
- `多套阅读主题`
- `更细的个性化排序`
- `本地搜索历史帖子`
- `收藏夹分组`

## 7. 明确不建议现在做的事

这些事情投入大、风险高，而且会把产品方向带偏。

- 发帖
- 回复
- 点赞
- 私信
- 完整消息中心
- 自动签到
- 高交互贴吧客户端能力

原因很直接：

- 这会显著增加登录、风控、接口稳定性、合规和维护成本
- 它们不是这个插件的核心差异化价值

## 8. 关键产品决策建议

### 8.1 `最新` 的处理建议

建议二选一：

1. 改名为 `最近查看`
2. 真做成“关注吧聚合流”

如果短期不准备做聚合流，优先建议改名。

### 8.2 `收藏` 的处理建议

建议也二选一：

1. 重新恢复为 `稍后看`
2. 完全降级成帖子页里的轻操作，不再作为主导航

如果目标是持续阅读，我更建议走 `稍后看`。

### 8.3 数据源策略建议

建议继续维持现在这个策略：

- `aiotieba bridge` 作为主路径
- 网页抓取作为 fallback

但要把“当前实际使用的是哪条路径”明确展示给用户。

## 9. 关键指标建议

即使现在还没接埋点，产品上也应该先明确看哪些指标。

- 首次成功打开帖子率
- 首次配置账号完成率
- fallback 触发率
- 每日平均打开帖子数
- 继续阅读使用率
- 老板模式触发率

这些指标会直接告诉你，问题到底出在：

- 首启门槛
- 数据稳定性
- 阅读体验
- 还是伪装场景

## 10. 建议的下一步执行顺序

建议按这个顺序推进，不要跳着做：

1. 先做 `环境诊断 + 状态可见`
2. 再做 `同步关注吧 + 继续阅读`
3. 再做 `稍后看 / 未读`
4. 最后再做 `老板模式增强`

原因是：

- 前两步解决“能不能稳定用”
- 第三步解决“会不会持续用”
- 第四步解决“是不是比浏览器更值”

## 11. 一句话结论

这个项目值得继续做，而且现在最该做的不是继续加零散按钮，而是补齐 `可用性闭环` 和 `持续阅读闭环`。

如果路线收得住，它会是一个很有辨识度的 VS Code 摸鱼阅读插件。
