# Tieba Fish 产品路线图

## 1. 文档目的

这份文档不再只回答“这个产品该做什么”，还要回答 3 个更实际的问题：

- 现在已经做到哪一步了
- 下一阶段到底先做什么
- 哪些事情明确不做

当前判断基于 2026-04-13 这版代码和交互现状。

## 2. 当前产品定义

Tieba Fish 的产品定位不应该是“完整贴吧客户端”，而应该稳定收敛为：

**一个常驻 VS Code 的贴吧阅读器，强调低打扰、结构化阅读、快速伪装和持续阅读。**

这个定位下，优先级顺序应该固定为：

1. 少切浏览器
2. 快速进入内容
3. 接着上次继续看
4. 需要时快速伪装

### 2.1 核心使用场景

后续所有需求都应该能落到下面 4 个实际场景之一：

- `碎片浏览`
  说明：工作间隙快速点开一个吧或帖子，看几层就收回去。
- `中断后继续阅读`
  说明：被打断后，下次回来不需要重新找帖子、页码和阅读位置。
- `低风险摸鱼`
  说明：界面存在感足够低，且需要时能快速切进老板模式。
- `弱环境下尽量可用`
  说明：即使登录态不完整、Python 不可用或接口偶发失败，也要尽量给出清晰 fallback。

## 3. 当前版本快照

### 3.1 已经成立的产品能力

当前已经具备这些核心能力：

- `aiotieba bridge` 作为结构化数据主路径
- 网页抓取 fallback
- 论坛页 / 帖子页 reader mode
- `关注吧 / 最新 / 历史` 原生 TreeView
- 浏览指定链接并在 Webview 打开
- 楼中楼按楼层展开
- 图片缩略图 + Webview 内大图预览
- 登录态导入
- 环境诊断页和状态栏运行状态提示
- 同步“我关注的贴吧”
- 老板模式

### 3.2 当前最有价值的差异化点

当前最有辨识度的点已经比较明确：

- `低打扰阅读`
- `结构化贴吧数据`
- `老板模式`

### 3.3 当前版本仍然是哪个阶段

现在的产品更接近：

- `可用 MVP`：是
- `可持续使用的阅读器`：部分成立
- `闭环完整产品`：还不是

原因不是功能太少，而是下面 3 个闭环还没完全打通：

1. 首次使用闭环
2. 持续阅读闭环
3. 信息架构闭环

### 3.4 近期已经收口的优化

这些点已经做完，不需要再反复回到“要不要做”的讨论：

- 命令展示去掉了冗余的 `Tieba:` 前缀
- `最新` 已通过空态文案和顶部说明解释真实语义
- `关注吧 / 最新 / 历史` 已有明确的 TreeView 加载反馈
- `最新` 的刷新、上一页、下一页已经有独立加载提示
- 楼中楼从“最多只看 3 条预览”升级为“展开后可分页浏览”

这些优化的价值不在于增加大功能，而在于降低等待时的不确定感和交互阻力。

## 4. 当前核心问题

### 4.1 首次使用门槛仍然偏高

虽然已经有环境诊断和状态提示，但最佳路径仍依赖：

- 本机 Python 可用
- `aiotieba` 已安装
- 用户知道如何从浏览器导入登录态
- 用户知道哪些功能需要 `STOKEN`

这意味着“可诊断”已经有了，但“足够傻瓜化”还没有。

### 4.2 `最新` 的语义已经收口，但仍需持续保持一致

当前 `最新` 已经通过界面文案解释为：

- 最近一次打开的关注吧的当前页快照

而不是：

- 关注吧聚合流
- 全局未读流
- 实时更新流

这个名字本身可以保留，但 README、诊断页和后续新入口也必须继续沿用同一套说法。

### 4.3 持续阅读链路还没有真正闭环

现在已经有：

- 历史记录
- 同步关注吧
- 页码级恢复

但还没有：

- 继续阅读入口
- 精确滚动位置恢复
- 楼层级恢复
- 稍后看
- 未读 / 已读

用户可以看，但还不够自然地“接着看”。

### 4.4 部分文档和产品现状仍不一致

例如：

- 实际主侧栏已经没有 `收藏`，但旧文档和部分认知还残留这个概念
- 登录态主交互已经是“导入登录态”，而不是“配置账号”
- `最新` 的命名、定位和真实能力还没统一

这些问题会直接拖慢后续迭代判断。

## 5. 已经确认的关键产品决策

这些事情现在已经不需要再反复讨论：

### 5.1 数据源策略

继续维持：

- `aiotieba bridge` 为主
- 网页抓取为 fallback

不要再回到“以网页嵌入或 Simple Browser 为主路径”的方案。

### 5.2 登录态策略

继续维持：

- 主交互是 `导入贴吧登录态`
- 推荐导入完整 Cookie
- 自动提取 `BDUSS / STOKEN`
- 网页 fallback 复用同一份 Cookie

不做：

- 账号密码登录
- 百度登录页重实现
- 二维码登录链路

原因很明确：`aiotieba` 本身不是这么设计的，而且在 VS Code 里做账号密码登录，成本高、风控重、产品可信度也差。

### 5.3 同步关注吧策略

第一版同步已经确定为：

- 手动触发
- 从贴吧账号单向导入到本地
- 不覆盖、不删除本地已有关注吧

这比“远端强覆盖本地”更稳，因为本地关注吧已经承担插件内订阅集合的角色。

### 5.4 老板模式策略

老板模式继续定位为：

- 快速伪装
- 低心智切换
- 优先保证“进得去、退得回”

先不追求：

- 100% 仿真 Explorer
- 完全恢复所有细粒度 UI 状态

### 5.5 需求评估原则

后续再进需求时，先用下面这 4 个问题判断值不值得做：

1. 这个需求是否能减少切去浏览器
2. 这个需求是否能让用户更容易接着上次继续看
3. 这个需求是否能降低等待和操作时的不确定感
4. 这个需求是否能强化老板模式或低打扰阅读的差异化

如果一个需求同时不满足上面 4 条里的至少 1 条，就不应该挤进近期主线。

### 5.6 版本推进闸门

每个版本在进入下一阶段前，至少要过下面的闸门：

- `V0.2 -> V0.3`
  要求：首启诊断清楚、登录态说明清楚、主路径和 fallback 可见、关键加载过程有反馈。
- `V0.3 -> V0.4`
  要求：继续阅读入口稳定、页码和滚动位置可恢复、历史和关注吧的边界清楚。
- `V0.4 继续扩展`
  要求：老板模式恢复稳定、伪装体验不打断主阅读链路、快捷键流足够顺手。

## 6. 版本状态与路线

### 6.1 V0.2 可用性版本

目标：解决“第一次装上能不能顺利用”的问题。

当前状态：`已做一半以上`

已经完成：

- 环境诊断页
- 状态栏运行状态提示
- 登录态导入
- 同步关注吧
- 文案从“配置账号”收口为“导入登录态”
- `最新` 说明文案补齐
- TreeView 和 `最新` 视图的基础加载反馈

仍未完成：

- 首次启动引导
- 对 `STOKEN` 作用的产品级说明
- 更统一的错误恢复提示

V0.2 验收标准：

- 用户第一次安装后，知道自己缺什么
- 用户知道当前是匿名还是登录态
- 用户知道当前走的是结构化数据还是网页 fallback
- 用户知道哪些功能需要完整登录态

### 6.2 V0.3 留存版本

目标：解决“看过一次后，为什么第二次还愿意继续用”的问题。

当前状态：`已开头，但远未完成`

已经完成：

- 历史记录
- 同步关注吧
- 页码级恢复的基础能力
- 楼中楼分页浏览

必须做：

- `继续阅读`
- 精确滚动位置恢复
- 帖子级最近阅读入口
- `稍后看`
- 阅读状态记录

建议做：

- 帖子未读标记
- 最近常看吧排序
- 打开时优先回到最近阅读内容

V0.3 验收标准：

- 用户不需要每次重新找帖子
- 用户可以自然地接着上次继续看
- 关注吧列表不依赖纯手工维护

### 6.3 V0.4 差异化版本

目标：把这个插件做成“只在 VS Code 里才成立”的摸鱼产品。

当前状态：`有亮点，但还不够稳`

已经完成：

- 老板模式基础切换
- 伪装文件树
- 伪装代码文件

必须做：

- 老板模式恢复更稳
- 快捷键阅读流
- 关键词静默过滤
- 更低存在感 UI 模式

建议做：

- 可配置伪装项目模板
- 恢复老板模式前的滚动位置
- 伪装状态下的假 Open Editors
- 自定义阅读快捷键

V0.4 验收标准：

- 产品在摸鱼场景下明显优于浏览器
- 用户可以不依赖鼠标完成核心浏览
- 老板模式进入和恢复都足够稳

## 7. 当前推荐排期

### 7.1 当前 Sprint 1

建议先把 V0.2 收尾，不要立刻开太多新坑。

P0：

- 首次启动引导页
- 登录态提示细化
- 同步关注吧成功后提示最近同步时间
- 论坛页 / 帖子页操作级加载反馈统一
- README / 命令 / 诊断页继续收口

验收：

- 新用户装上后不需要猜当前状态
- “同步关注吧”与“导入登录态”的关系清晰

### 7.2 当前 Sprint 2

开始补持续阅读闭环。

P0：

- `继续阅读`
- 恢复上次阅读帖子
- 记录并恢复滚动位置
- 最近阅读吧排序
- 楼中楼分页状态记忆

P1：

- 稍后看
- 已读 / 未读

验收：

- 第二次打开扩展时，不需要重新找刚看过的帖子

### 7.3 当前 Sprint 3

开始把差异化能力做稳。

P0：

- 老板模式恢复稳定性
- 帖子快捷键导航
- 关键词静默过滤

P1：

- 假 Open Editors
- 更细的伪装模板

验收：

- 用户能明显感知到它比浏览器更适合摸鱼

### 7.4 当前两周内最值得继续收口的点

如果近期只继续推进 4 件事，我建议固定为：

1. 首次启动引导
2. 登录态提示细化
3. `继续阅读`
4. 滚动位置恢复

原因：

- 前两项解决“为什么这次能用 / 不能用”
- 后两项解决“为什么下次还愿意回来继续看”

## 8. Backlog 优先级

### 8.1 P0

这些是现在最值得立即排期的项。

- `最新` 说明文案补齐
  说明：保留现有名字，但把“当前来自哪个吧、显示的是什么数据”讲清楚。
- `首次启动引导`
  说明：把 Python、aiotieba、登录态、STOKEN 的概念解释清楚。
- `继续阅读`
  说明：给出明确入口，恢复最近帖子和页码。
- `滚动位置恢复`
  说明：把“继续阅读”真正做实。
- `最近同步时间`
  说明：让同步关注吧更可见。
- `加载反馈统一`
  说明：把 TreeView、论坛页、帖子页、楼中楼的加载反馈收成一致规则。
- `文档和界面统一`
  说明：继续消除命令、README、界面表述不一致。

### 8.2 P1

这些会明显提升体验，但不是当前阻塞项。

- `稍后看`
- `未读 / 已读`
- `关键词过滤`
- `状态栏快捷操作入口`
- `帖子内导航快捷键`
- `阅读来源提示`
- `最近常看吧排序`
- `楼中楼分页状态记忆`

### 8.3 P2

这些可以做，但不应该现在优先。

- `更复杂的伪装模板`
- `多套阅读主题`
- `更细的个性化排序`
- `本地搜索历史帖子`
- `稍后看分组`

### 8.4 候选优化池

这些需求值得继续观察，但先不直接插进 P0：

- `紧凑阅读模式`
  说明：进一步降低论坛页和帖子页的视觉存在感。
- `局部失败重试`
  说明：楼中楼、帖子翻页、最新翻页失败时可原地重试，而不是整页重开。
- `阅读来源提示`
  说明：更明确地区分 `aiotieba`、网页 fallback 和缓存。
- `最近常看吧排序`
  说明：让高频贴吧更容易点到，但不要过早做复杂推荐。
- `老板模式假 Open Editors`
  说明：拟真度更强，但应放在恢复稳定之后。

## 9. 需要特别关注的风险

### 9.1 外部依赖风险

- `aiotieba` 依赖 Python 运行时
- 本地未安装依赖时体验会下降
- 百度接口和风控可能随时变化

### 9.2 登录态风险

- 用户不理解 `BDUSS` 和 `STOKEN` 的区别
- 只导入 `BDUSS` 时，部分功能会“能看但不同步”
- 登录态失效后，错误提示如果不够明确，用户会以为插件坏了

### 9.3 信息架构风险

- `最新` 如果长期缺少解释文案，仍会持续误导
- 外部文档和新入口如果重新用回“全局最新”的说法，会把语义再次带偏
- `历史`、`最新`、`稍后看` 未来容易出现边界重叠

## 10. 明确不建议现在做的事

这些事情投入大、风险高，而且会把产品方向带偏。

- 发帖
- 回复
- 点赞
- 私信
- 完整消息中心
- 自动签到
- 高交互贴吧客户端能力

原因：

- 会显著增加登录、风控、接口稳定性、合规和维护成本
- 它们不是这个插件当前最核心的差异化价值

## 11. 关键指标建议

即使现在还没正式接埋点，产品上也应该先明确看哪些指标。

- 首次成功打开帖子率
- 首次导入登录态完成率
- fallback 触发率
- 同步关注吧使用率
- 每日平均打开帖子数
- 继续阅读使用率
- 老板模式触发率

这些指标会直接告诉你问题到底出在：

- 首启门槛
- 数据稳定性
- 阅读体验
- 还是伪装场景

## 12. 一句话结论

这个项目值得继续做，而且当前最应该做的不是继续堆零散按钮，而是：

- 收尾 `可用性闭环`
- 补齐 `持续阅读闭环`
- 再把 `老板模式` 做成真正的差异化能力

如果路线收得住，它会是一个有明确辨识度的 VS Code 摸鱼阅读插件。
